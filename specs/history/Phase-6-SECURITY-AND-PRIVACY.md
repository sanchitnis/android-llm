# Security, Data Privacy and Sandboxing - Admission Counselor AI

This document establishes the storage sandboxing paths, local database encryption parameters, transcript privacy rules, and Android permission containment configurations to secure student data.

---

## 1. Application Storage Sandboxing

The application executes within the standard Android sandboxed environment. All application assets, database records, and inference models are stored exclusively in internal private directories.

### 1.1 Sandbox Directories Mapping

| Data Target | File Path Location | Access Restrictions |
| :--- | :--- | :--- |
| **Model Asset** | `/data/user/0/com.admission.counselor/files/` | Read-only access by the LiteRT-LM engine. Access blocked for other applications. |
| **Admission Database** | `/data/user/0/com.admission.counselor/databases/` | Read/write access restricted to the local database manager helper. |
| **Keystore Key Material** | `/data/user/0/com.admission.counselor/shared_prefs/` | Managed via `EncryptedSharedPreferences` backed by the Android Keystore. |

## 2. Local Storage Encryption

To prevent extraction of university guidelines or criteria from the database via rooted devices, all persistent storage is encrypted at rest.

### 2.1 SQLite Encryption with SQLCipher
The local database `admission.db` is encrypted using SQLCipher. 
1. **Algorithm**: AES-256-GCM.
2. **Key Storage**: The database encryption key is generated at first launch, stored in the Android Keystore, and retrieved at runtime using the Android Jetpack Security library.
3. **Connection Hook**:
   ```kotlin
   val factory = SupportFactory(SQLiteDatabase.getBytes(passphrase))
   val database = Room.databaseBuilder(context, AdmissionDatabase::class.java, "admission.db")
       .openHelperFactory(factory)
       .build()
   ```

## 3. Data Privacy and Transcript Policy

To protect student queries, GPA disclosures, and personal background details:

### 3.1 In-Memory Transcript Rule (Resolution of ISS-006)
- **Zero Persistent Chat Logs**: Conversation histories are stored **exclusively in-memory** using Kotlin StateFlow.
- **Volatile Lifetimes**: The transcript is never saved to SQLite, SharedPreferences, or external files.
- **Automatic Destruction**: The conversation log is wiped from memory under three events:
  1. The user terminates the chat session.
  2. The 5-minute idle inactivity timeout triggers a model unload.
  3. The OS triggers a critical low-memory (`TRIM_MEMORY_RUNNING_CRITICAL`) event.

### 3.2 Telemetry and Analytics Policy
- **Zero Telemetry**: All third-party telemetry, usage tracking, and performance analytics tools (e.g., Firebase Analytics, Crashlytics) are completely disabled and omitted from compilation.
- **No Diagnostics Leak**: Internal error logs generated by the LiteRT-LM runtime are routed to the local Android Logcat buffer and are never transmitted to external cloud systems.

## 4. Zero Network Permission Model

To guarantee that no student data can leave the device boundary, the application is prohibited from opening socket connections.

### 4.1 Permission Manifest Rule
The application's `AndroidManifest.xml` file must contain zero network-related permission definitions.

```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.admission.counselor">
    
    <!-- 
      The following permissions are STRICTLY FORBIDDEN and must not be declared:
      - android.permission.INTERNET
      - android.permission.ACCESS_NETWORK_STATE
      - android.permission.ACCESS_WIFI_STATE
    -->
    
    <application
        android:allowBackup="false"
        android:theme="@style/Theme.AdmissionCounselor">
        <!-- Application Components -->
    </application>
</manifest>
```

### 4.2 Network Isolation Impact

| Benefit | Trade-off | Mitigation |
| :--- | :--- | :--- |
| Guaranteed 0% chance of user query data leaking to external networks. | App cannot download model updates or database criteria updates dynamically. | Model updates and database changes must be packaged into standard app updates shipped through Google Play. |
| Eliminates network latency bottlenecks and battery usage from network polling. | Initial app install size is larger if the model is bundled in the split APKs. | Managed via Play Asset Delivery (PAD) as detailed in Phase 7. |
