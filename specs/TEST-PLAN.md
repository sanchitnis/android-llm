# Test & Verification Plan - Admission Counselor AI

Last Modified: June 18, 2026

This document details the automated testing strategies, performance benchmarking setups, memory profiling procedures, and on-device Retrieval-Augmented Generation (RAG) verification targets for the Admission Counselor App.

---

## 1. Automated Testing Strategy

The application operates in a completely offline environment without internet access (`android.permission.INTERNET` is omitted). Consequently, all testing strategies focus on local, on-device execution and mock dependencies.

```
+-------------------------------------------------------------+
| UI / View Testing (Compose UI Tests & Espresso)              |
+-------------------------------------------------------------+
                            |
                            v
+-------------------------------------------------------------+
| ViewModel / Business Logic (Robolectric & JUnit)            |
+-------------------------------------------------------------+
                            |
                            v
+-------------------------------------------------------------+
| Local Engines & RAG DB (SQLCipher & LiteRT-LM Mock Runtime)  |
+-------------------------------------------------------------+
```

### 1.1 Automated Test Suites

| Test Category | Frameworks Used | Execution Environment | Test Objective |
| :--- | :--- | :--- | :--- |
| **Unit Tests** | JUnit 4, Mockk, Robolectric | Local JVM / Android Emulator | Verify isolated business logic, ViewModel state transitions, local database queries, and concurrency locks. |
| **Integration Tests** | AndroidX Test, Espresso, Compose UI Test | Android Emulator / Physical Device | Validate the interplay between the ViewModel, RAG Retrieval Manager, Play Asset Delivery triggers, and Local Llm Engine. |
| **Performance Benchmarks** | Jetpack Microbenchmark / Macrobenchmark | Physical Reference Device (e.g., Pixel 7) | Measure time-to-first-token (TTFT), generation throughput, and memory ceilings under sustained load. |

## 2. Unit Testing Cases

Unit tests target components in isolation. External system dependencies, such as the Android OS memory trimmer and the actual LiteRT binary, are mocked or simulated.

### 2.1 ViewModel and UI State Transition Tests

These tests verify that `CounselorViewModel` responds correctly to user interactions and updates the UI StateFlow.

| Test ID | Target Component | Input / Action | Mock Setup | Expected Result |
| :--- | :--- | :--- | :--- | :--- |
| **UT-VM-001** | `CounselorViewModel` | User sends first message when model is unloaded. | `LocalLlmEngine` mocked to return lazy initialization status. | StateFlow transitions from `Idle` to `LoadingModel`, then to `Generating` once the stream begins. |
| **UT-VM-002** | `CounselorViewModel` | User sends message while engine is busy. | `LocalLlmEngine` is mock-configured to throw `EngineBusyException` or return busy status. | StateFlow emits warning state without enqueueing request; a busy banner is displayed. |
| **UT-VM-003** | `CounselorViewModel` | Model initialization fails. | `LocalLlmEngine` throws `InitializationException` on load. | StateFlow transitions to `Error` state; UI shows Retry button. |

### 2.2 Database and Cryptography Setup Tests

These tests verify database copying, SQLCipher decryption, and database schemas.

| Test ID | Target Component | Input / Action | Mock Setup | Expected Result |
| :--- | :--- | :--- | :--- | :--- |
| **UT-DB-001** | `DbProvisioningManager` | First launch of application. | Mock Assets to contain template `admission.db`. | File copied to `/data/user/0/com.admission.counselor/databases/admission.db`, decrypted using the key generated by EncryptedSharedPreferences, and verified readable. |
| **UT-DB-002** | `RAGRetrievalManager` | Query "BTech Tuition Fee". | Database populated with mock course fees. | Returns exact database records for BTech tuition fees with relevancy scores. |
| **UT-DB-003** | `DbProvisioningManager` | Re-launch when database already exists. | File exists in target path. | Skipping copy logic; open existing database successfully. |

### 2.3 Mutex and Thread Containment Tests

These tests ensure that parallel calls to the inference engine are blocked by the Mutex and execute sequentially on the dedicated thread.

| Test ID | Target Component | Input / Action | Mock Setup | Expected Result |
| :--- | :--- | :--- | :--- | :--- |
| **UT-MTX-001** | `LocalLlmEngine` | Trigger two concurrent `generateResponse` calls in separate coroutines. | Slow generation mock (each taking 2 seconds). | First request acquires lock and generates tokens; second request is immediately rejected with `EngineBusyException` (non-queuing busy policy). |
| **UT-MTX-002** | `LocalLlmEngine` | Execute engine load and unload calls. | Log thread details during execution. | All engine operations run on the designated single thread (`LlmThread`) and do not block `Dispatchers.Main`. |

## 4. Performance & Latency Benchmarks

Performance tests validate the non-functional requirements (NFRs) defined in the PRD. These tests must run on physical reference devices under controlled environments (disabled thermal throttling where possible, standard ambient temperature, and consistent battery levels).

### 4.1 Target Latency & Throughput Benchmarks

| Metric Profile | Target Accelerator | Input Context Size | Target Benchmark | Measurement Method |
| :--- | :--- | :--- | :--- | :--- |
| **Time to First Token (TTFT)** | GPU (ML Drift) | 1000 tokens | Under 0.5 seconds | Jetpack Macrobenchmark measuring time from query submit to first character emitted. |
| **Time to First Token (TTFT)** | CPU (XNNPACK) | 1000 tokens | Under 1.5 seconds | Jetpack Macrobenchmark measuring time from query submit to first character emitted. |
| **Generation Throughput** | GPU (ML Drift) | 1000 tokens | 15 tokens per second or higher | Calculated as total generated tokens divided by generation duration (excluding TTFT). |
| **Generation Throughput** | CPU (XNNPACK) | 1000 tokens | 5 tokens per second or higher | Calculated as total generated tokens divided by generation duration (excluding TTFT). |

### 4.2 Resource and Thermal Benchmarks

| Benchmark ID | Test Scenario | Measurement Target | Budget limit | Expected Outcome |
| :--- | :--- | :--- | :--- | :--- |
| **PB-RES-001** | Continuous generation (300 tokens). | Peak RAM footprint. | Under 2.0 GB | System RAM allocation does not exceed 2.0 GB, verified via Android Profiler. |
| **PB-THR-001** | Sustained chat conversation (continuous prompts for 3 minutes). | Generation throughput degradation. | Under 20% degradation | Initial throughput of 15 tokens/sec does not drop below 12 tokens/sec, preventing severe performance dips. |
| **PB-THR-002** | Device temperature reaches 42 degrees Celsius (Warm). | Pause delay injection. | 40ms token pause | System detects Warm state via HardwarePropertiesManager, injects 40ms pauses, and prevents temperature from reaching the Throttled limit. |
| **PB-THR-003** | Device temperature reaches 46 degrees Celsius (Throttled). | Fallback delegate activation. | Switch to CPU | System disables speculative decoding and falls back to CPU delegate, reducing thermal load. |

## 5. Memory Leak & Low-Memory (LOMEM) Profiling

Memory tests ensure the application does not trigger Out-of-Memory (OOM) crashes and releases the 2.58 GB model asset when required.

### 5.1 Memory Profiling Scenarios

| Test ID | State / Trigger | Test Procedure | Expected System Action | Verification Method |
| :--- | :--- | :--- | :--- | :--- |
| **MP-LIM-001** | 5-Minute Inactivity | 1. Initialize model.<br>2. Wait 300 seconds without inputs. | Engine closes `LlmInference` instance, nullifies pointer, and calls `System.gc()`. | Verify RAM footprint drops from ~2.0 GB back to baseline app RAM (~80 MB) using Android Studio Memory Profiler. |
| **MP-LIM-002** | App sent to background | 1. Initialize model.<br>2. Move app to background (ON_STOP event). | Release model memory immediately. | Verify RAM footprint drops immediately to baseline app RAM. |
| **MP-LIM-003** | Critical low-memory signal | 1. Set engine state to active generation.<br>2. Force `TRIM_MEMORY_RUNNING_CRITICAL` via ADB. | 1. Terminate generation stream.<br>2. Close engine instance.<br>3. Release memory. | Verify session terminates cleanly without crashing, and RAM is reclaimed. |
| **MP-LIM-004** | Moderate memory pressure | 1. Active conversation history filled.<br>2. Trigger `TRIM_MEMORY_RUNNING_MODERATE`. | Conversation history context in VM is truncated to the last 2 turns. | Verify context tokens sent to model are reduced. |
| **MP-LIM-005** | High memory pressure | 1. Active speculative decoding running.<br>2. Trigger `TRIM_MEMORY_RUNNING_LOW`. | 1. Unload drafter model (150 MB).<br>2. Disable speculative decoding. | Verify drafter model reference is cleared and speculative decoding is inactive. |

### 5.2 ADB Commands for Simulation

To execute low-memory simulations manually, developers run the following ADB commands:

```bash
# Simulate TRIM_MEMORY_RUNNING_CRITICAL (Level 80)
adb shell am send-trim-memory com.admission.counselor RUNNING_CRITICAL

# Simulate TRIM_MEMORY_RUNNING_LOW (Level 60)
adb shell am send-trim-memory com.admission.counselor RUNNING_LOW

# Simulate TRIM_MEMORY_RUNNING_MODERATE (Level 5)
adb shell am send-trim-memory com.admission.counselor RUNNING_MODERATE
```

## 6. On-Device RAG Accuracy & Verification Targets

These tests verify that context retrieval matches queries, and the model formats answers correctly within the 4096-token context window budget.

### 6.1 RAG and Prompt Verification Table

| Test Case Category | Query Input | Injected Context | Target Response Requirement | Pass Criteria |
| :--- | :--- | :--- | :--- | :--- |
| **Context Inclusion** | "What is the BTech fee?" | "BTech Tuition Fee is 150,000 INR per year." | Provide the tuition fee for the BTech course. | Response states "150,000 INR per year" exactly. |
| **Context Exclusion** | "What is the fee for MBA?" | "BTech Tuition Fee is 150,000 INR per year." (No MBA fee in DB). | State that MBA tuition fees are not available offline. | Response states that information is not available and does not hallucinate fees. |
| ** hallo-mitigation** | "Do you offer Aerospace engineering?" | Empty context returned due to low score. | Inform student that Aerospace engineering is not listed. | Response does not claim the course is offered. |
| **Hallucination Check** | "Tell me about course X." | Course X context. | Answer using *only* the provided context block. | Response does not contain external web facts. |
| **Constraint Budget** | Extremely long history + long context + query. | 1200 tokens of context + 800 tokens of history. | Truncate history to prevent exceeding 4096 context budget. | System does not throw `ContextWindowExceededException` or fail inference. |
